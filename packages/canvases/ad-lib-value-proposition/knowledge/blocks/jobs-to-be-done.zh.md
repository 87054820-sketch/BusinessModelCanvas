客户雇用某个方案要完成的任务，可以是功能性、社会性或情感性的。用「动词 + 结果」表达：「按时交付一个功能」，而不是「需要一个工具」。

## 模块作用

这个槽位解释客户想取得的进展，而不是你的解决方案。它应把访谈证据或 JTBD 故事压缩成一个短结果。

## 好答案 vs 弱答案

好答案采用「动词 + 结果」，即使你的产品不存在也成立。弱答案描述功能，例如「使用 AI 看板」，而不是客户想完成的进展。

## 常见误区

不要把任务误写成特性需求。「导出报表」可能只是操作；「不用手工对账也能向 CFO 解释表现」更接近工作。

## 便签示例

- 「不用追五张表也能完成月结。」
- 「在不冒采购失误风险的情况下选择供应商。」

## 迁移问题

- 客户想让自己的工作或生活发生什么进展？
- 这个工作变得紧急之前发生了什么？
- 如果你的产品消失，这个工作是否仍然重要？

## 来源依据

Value Proposition Design 的 Jobs/Pains/Gains 与 JTBD 方法都要求先描述客户进展，再描述方案。这个槽位保留了这个顺序。
