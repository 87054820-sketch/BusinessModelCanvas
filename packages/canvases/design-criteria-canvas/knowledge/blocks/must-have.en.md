# Must Haves

These are non-negotiable design criteria. If an idea cannot satisfy these standards, it should not move forward.

## How to write

- Start with “Must…” and make the boundary clear.
- Prefer testable criteria over wishes.
- Ask: would the project truly fail if this criterion were missing?

## Examples

- “Must validate real purchase intent within three months”
- “Must comply with current security and data rules”
- “Must let core users reach first value in under 10 minutes”

## Good Answer vs Weak Answer

A good Must Have is a true gate: if it fails, the solution should stop, pivot, or return to discovery. A weak answer is an executive preference or nice feature promoted to "must" because the team wants certainty.

## Common Pitfalls

Do not overload Must Haves until every idea fails. Also do not write vague criteria such as "must be user friendly"; define the evidence or threshold that proves it.

## Sticky Examples

- "Must show purchase intent from at least 5 target customers before build funding."
- "Must meet SOC2-compatible data handling before any enterprise pilot."

## Transfer Questions

- Would we kill or redesign the idea if this criterion is not met?
- What evidence proves the criterion has been met?
- Is this a customer, business, ethical, or operational non-negotiable?

## Source Grounding

Design A Better Business frames design criteria as standards for judging concepts. Value Proposition Design and Testing Business Ideas add the discipline of connecting must-have criteria to customer evidence and validation thresholds.
