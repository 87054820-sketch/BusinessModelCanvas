# Should Haves

These criteria are important but not absolute deal breakers. They usually have a strong effect on user experience, business viability, or organizational adoption.

## How to write

- Include standards the team strongly wants to satisfy.
- If resources are tight, these can be reduced or delayed.
- Explain why the criterion matters and what is lost if it is missing.

## Examples

- “Should reuse existing sales channels”
- “Should support a mobile-first experience”
- “Should be configurable by the support team”

## Good Answer vs Weak Answer

A good Should Have explains why the criterion matters and what trade-off is acceptable if it is delayed. A weak answer hides a Must Have in softer language or lists broad preferences without decision impact.

## Common Pitfalls

Do not let Should Haves quietly become Must Haves during review. If missing the criterion would truly stop the project, promote it. If not, define what quality level is acceptable for the next iteration.

## Sticky Examples

- "Should reuse the existing partner channel if CAC stays below the direct-sales benchmark."
- "Should support mobile onboarding for field users, but desktop pilot is acceptable for first validation."

## Transfer Questions

- What becomes worse if this criterion is missing, and is that acceptable?
- Which Must Have does this support?
- When should it be revisited?

## Source Grounding

MoSCoW-style prioritization distinguishes critical requirements from important but negotiable ones. The Design Criteria Canvas applies that logic to business model and experience design decisions, not only feature ranking.
