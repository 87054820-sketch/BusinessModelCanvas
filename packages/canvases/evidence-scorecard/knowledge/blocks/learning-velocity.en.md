# Learning Velocity

Learning velocity is the pace at which tests produce decision-changing insight. It is not the number of experiments completed. A team can run many shallow tests and still learn slowly; another team can run one sharp pre-sale and learn enough to change the roadmap.

## Good Answer vs Weak Answer

A good answer names the cadence, the latest learning, and what changed because of it. A weak answer lists activity volume: "three interviews, one survey, one prototype." The question is whether the team is reducing uncertainty fast enough.

## Common Pitfalls

- Rewarding busyness instead of learning.
- Running the same low-strength test repeatedly.
- Failing to update assumptions, BMC blocks, or next experiments after learning.

## Sticky Examples

- "One decision-changing test every 10 working days; last test moved buyer from user to VP Ops."
- "Velocity low: four weeks spent on prototype polish, no new customer behavior."

## Transfer Questions

- What did the team believe last month that it no longer believes?
- Which decision has been unlocked by the latest learning?
