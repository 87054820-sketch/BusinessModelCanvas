# 使用方法

1. **定义焦点议题。** 写清情景要服务于哪个战略决策、商业模式、组合或市场问题。
2. **从环境信号开始。** 用 `business-model-environment` 收集趋势、市场力量、行业力量、宏观力量和弱信号。
3. **区分趋势和不确定性。** 趋势可能影响所有未来；关键不确定性会创造不同未来。
4. **选择两个相互独立的不确定性。** 每个不确定性都应显著改变战略选择，并有两个可信端点。
5. **构造四个情景。** 每个情景都应描述市场结构、客户行为、竞争动态、运营约束和 BMC 压力点。
6. **做风洞测试。** 把当前 BMC、新业务选项、组合押注和设计准则放进每个情景中检验。
7. **记录稳健动作和早期信号。** 稳健动作能跨多个未来成立；早期信号提示哪个未来可能正在形成。
8. **如果这是组织项目，要做评估。** 记录决策、假设和后续监测发生了什么变化。

# 如何解读

强情景矩阵会改变团队要做的事。如果每个情景最后都导向同一个泛泛战略，通常说明不确定性太弱，或情景含义没有展开。

# 质量检查

- 焦点议题明确。
- 两个不确定性相互独立。
- 每个情景有不同战略逻辑，而不只是不同情绪。
- 每个情景至少压力测试一个 BMC 模块或组合选择。
- 稳健动作和早期信号可执行。
- 后续监测有负责人。

# 反模式

- 把四个情景写成乐观、中性、悲观加灾难版。
- 情景很生动，但没有改变任何战略选择。
- 两个不确定性其实描述的是同一股力量。
- 忽略早期信号，导致情景规划无法被持续更新。
- 把工作坊产物当终点，而不是把情景部署到决策中。

# 输入和输出

输入包括焦点决策、环境信号、趋势、潜在不确定性、当前 BMC 假设，以及正在考虑的战略选项。输出包括四个被命名的未来、对商业模式的影响、需要监测的早期信号，以及在多个未来中都仍然有价值的稳健动作。

# 相关学习路径

先读 *The Art of the Long View* 中关于情景逻辑、驱动力、风洞测试、战略含义和早期指标的章节。如果情景要成为组织里的持续机制，而不是一次工作坊，再读 *Scenario Planning in Organizations*。HBR 的不确定性战略框架适合帮助判断什么时候应该用情景，而不是单一预测。
