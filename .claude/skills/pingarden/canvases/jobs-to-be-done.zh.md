# JTBD 工作故事

## 何时使用

当"用户需求"或"persona"太模糊,推不动产品决策时,用 Jobs-to-Be-Done(JTBD)。JTBD 问的是:**客户在"雇佣"你的产品做什么?** 它把分析单位从"谁"换成"想取得什么进展",并强制团队把这个进展写成一个故事:情境 → 动机 → 理想结果(功能性 + 情感 + 社会三层)。

这是 5 块顺序画布,一张图对应一个工作。多个工作 = 多张图。

## 何时不该用

- 多细分业务设计 —— 那是 **BMC**。
- 描述聚合 persona(JTBD 关心的是某个具体时刻的进展,不是这个人整体是谁)—— 需要 persona 锚点的话用 **Empathy Map**。
- 构建提供物本身 —— JTBD 给 **VPC** / **Design Criteria Canvas** 供料。

## 填写顺序 —— 一个故事,顺序填,不能跳

1. **`situation`** —— 这个工作什么时候冒出来?具体情境:几点钟、刚发生了什么、他们之前在做什么。不是人口统计("谁"),而是何时和正在做什么。
2. **`motivation`** —— 为什么 _这个时刻_ 这个工作变得重要?是什么压力让它浮出来?(截止日期 / 出错 / 新机会 / 身份焦虑 / 等等)
3. **`ideal-outcome`** —— 功能层面的终点长什么样?具体且可衡量:"周五 6 点前把报告交出去,所有数字对得上"。不是"提高效率"。
4. **`emotional-social`** —— 这件事让他们**感觉**怎样(松一口气 / 自豪 / 避免羞愧)?在别人眼里**显得**怎样(有能力 / 认真 / 掌控感)?功能性工作总是套在情感和社会的外壳里 —— 忽略这层就是为什么"理性"功能上线后没人买账。

每一块都必须建立在前一块之上。如果 `motivation` 不能从 `situation` 自然流出来,你描述的是两个不同的工作。

## 跨模块一致性

- **`motivation`** 必须由 `situation` 触发 —— 不是泛泛的偏好。
- **`ideal-outcome`** 必须是可衡量的终点状态,不是"省时间"或"高效"。
- **`emotional-social`** 必须解释这个功能性结果在社会 / 情感上为什么**重要** —— 失败的话客户会怕什么?
- 整张画布应该读起来像一个故事。如果你不能用一句话讲完("当 _<情境>_,我想要 _<动机>_,这样我可以 _<结果>_,这样我感觉 _<情感/社会>_"),这张画布就没填完。

## 反模式 —— 不允许

- ❌ **把工作写成功能。** "用 AI 总结会议记录"是功能。工作是"下一次 1:1 不用重听录音也显得有备而来"。
- ❌ **只写功能层,忽略情感 / 社会。** B2B 的 JTBD 失败大多来自漏掉社会层 ——"在审计员面前显得认真" —— 而这往往才是真正驱动决策的东西。
- ❌ **`situation` 写成 persona。** "28-35 岁中级市场经理"是 persona。情境是一个瞬间:"在第二次提报被退回之后"。
- ❌ **一张图多个工作。** 如果你的 `motivation` 能往两个方向走,拆。
- ❌ **结果写成"用上产品"。** "装上了 AI 工具"是拥有产品,不是结果。结果是这个 AI 工具在他们的世界里**产出**了什么。
- ❌ **忽略失败模式。** 真正重要的工作都有"如果搞砸了"的阴影。如果失败感觉无所谓,这个工作不值得为它做设计。

## 语气

整张画布读起来要像一个故事,不是一个清单。每一块 1-3 句话内 —— 是一个素描,不是一份立项书。

## Blocks

`zoneId` 是 API 的稳定标识符,`pingarden canvas write` payload 必须用它,不要翻译。

### `situation` — When(当...)

**Prompt** — 客户当下处于什么情境?是什么刚刚发生的事触发了这个需求?

**Example** — 周五下午 4 点,CFO 刚要求周一前给出下一季度的预算偏差报告

**Quality bar** — 每条工作故事的最左一格。 Christensen 反复强调:**情境对工作的定义是 根本性的**。 没有情境,就不是「工作」,而是一份功能说明。 刚要求周一前给出预算偏差报告」好过「需要做报表」。 时间和地点决定了「足够好」的尺子。

### `motivation` — I want to(我想...)

**Prompt** — 客户想做什么?以动词开头。(可以是功能性、情感性或社交性的。)

**Example** — 把每周做报表的时间从 6 小时压到 1 小时以内(功能性)

**Quality bar** — 中间这一格。 客户在那个情境下想做的**动作**。 开头一定是动词。

### `ideal-outcome` — So I can(从而我能...)

**Prompt** — 客户在追求什么样的「进展」?把结果写成动作动词,不是被动状态。

**Example** — 从五家供应商里挑出值得回电话的两家

**Quality bar** — 最右一格。 客户想取得的**进展**。 Christensen 用的词是「progress(进展)」: Make Us Proud 团队的修订版强调:**结果要以动作动词写出来**,不要写 成被动状态。

### `emotional-social` — 情感 + 社交任务

**Prompt** — 针对上面每一条功能性工作故事,至少为它配一条情感任务和一条社交任务。

**Example** — 情感 — 月结那一周心定下来,不再焦虑

**Quality bar** — 底部这一行。 **上面每一条功能性工作故事,至少要在这里配一条情感任务 和一条社交任务**。 配不上,你拿到的就只是半张图。 客户买产品不只是为了把事情搞定 —— 他们也在买那种感受,以及别人怎么 看待自己。

## Colour legend

- `0` — **功能性任务**: 客户想完成的实际任务、目标或问题。
- `1` — **情感性任务**: 客户在任务前、中、后想获得或避免的情绪感受。
- `2` — **社交性任务**: 客户希望被他人如何看待。

## Pairs with

这张画布常和这些一起用,做完后引导用户接下去做:

- `value-proposition-canvas` — 把这条验证过的工作放进 VPC 中心,绕着它设计痛点缓解和收益创造。
- `empathy-map` — 把这个工作锚到一个具体 persona × 时刻,让它具体起来。
- `ad-lib-value-proposition` — 把验证过的工作 + 结果压缩进 Ad-Lib 一句话,用来对齐利益相关方。
- `customer-journey` — 标出这个工作在旅程的哪个阶段变得紧迫 —— 那就是产品该发力的地方。
