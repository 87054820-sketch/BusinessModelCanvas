# 价值主张画布

## 何时使用

价值主张画布(VPC)用于深入理解 **一个客户细分**,并设计价值主张如何契合他们的真实处境。BMC 里"客户细分 + 价值主张"那对只是勾画;VPC 是把那对放大到一页纸里看清楚。

**一张 VPC 对应一个客户细分。** 三个细分就要做三张 VPC。

## 何时不该用

- 跨细分的业务设计 —— 那是 BMC 的事。
- 泛泛的"头脑风暴" —— VPC 必须基于观察到的客户事实,不是空想。
- 给市场传达已经验证过的价值主张 —— 那是营销文案,不是画布。

## 填写顺序 —— 一定先填客户档案(右),再填价值地图(左)

最常见的翻车方式是先填价值地图;然后你会下意识把"痛点 / 收益"反推回你已有的产品。VPC 整张画布的意义就是 **抵抗** 这种反推。务必:

### 右侧 —— 客户档案(圆形):

1. **客户工作(Customer Jobs)** —— 客户想完成什么?区分:
   - _功能性_ 工作(执行任务:"申报年度增值税")
   - _社交性_ 工作(给别人留下印象:"显得对工作很有掌控感")
   - _情感性_ 工作(让自己感觉好:"在年终审计前安心")
   - 每条注明:发生情境、是关键还是次要。
2. **痛点(Pains)** —— 在尝试完成工作 _之前 / 中 / 后_ 让客户难受的任何东西:
   - 不想要的结果 / 问题 / 体验
   - 阻碍他们开始的障碍
   - 他们害怕的风险(频率多高?后果多重?)
3. **收益(Gains)** —— 他们 _想要_ 的成果。分四档:
   - _必需_ 收益(没有就完全不工作)
   - _预期_ 收益(基本期望,但不是死板必要)
   - _期望_ 收益(锦上添花)
   - _意外_ 收益(惊喜 —— 他们不会主动提)

### 左侧 —— 价值地图(方形):

4. **产品和服务(Products & Services)** —— 真正的"提供物"(不是空话,不是承诺)。盒子里到底是什么?SaaS 订阅里能用什么?
5. **痛点缓解(Pain Relievers)** —— 对每个痛点,你的提供物 _怎样_ 缓解或消除它?要逐条对应。
6. **收益创造(Gain Creators)** —— 对每个收益,你的提供物 _怎样_ 产生或放大它?要逐条对应。

### Bain 价值透镜(辅助,不是主画布):

当收益写得太泛时,用 `bain-elements-of-value` 给 `gains` / `gain-creators` 加轻量标签,例如 `[功能-节省时间]`、`[情感-降低焦虑]`、`[社会-身份认同]`。标签只能解释 VPC 的便签,不能替代真实的客户任务、痛点和收益。

## 跨模块一致性 —— VPC 的全部逻辑依赖这几条

- **每个痛点缓解应映射到 ≥ 1 个具体痛点。** 不对应任何真实痛点的"缓解"就是无病呻吟的功能。
- **每个收益创造应映射到 ≥ 1 个具体收益。** 同上。
- **关键痛点**(高频 / 严重)应该有 _更强_ 的缓解,而不是只用"也能解决"打发。
- **必需收益** 必须能可靠创造 —— 缺它,客户根本不会考虑你。
- **产品和服务** 是 _执行_ 缓解和创造的实物 —— 不是抽象的"价值"。

VPC 达成 _契合(fit)_ 的标志是:左侧真正命中右侧最重要的几条。**契合,而非对称数量,才是目标。**

## 反模式 —— 不允许

- ❌ **先填价值地图。** 然后你会写出"恰好"匹配现有产品的痛点和收益。客户档案必须先写,而且要假装你的产品不存在。
- ❌ **一张 VPC 服务多个细分。** "中小企业老板和大企业 CFO"—— 拆开。他们的工作 / 痛点 / 收益不同。
- ❌ **痛点写成"反向收益"、收益写成"反向痛点"。** 二者是不同维度。"省时间"是收益;"浪费时间"是痛点。同一句话翻面,不算两边都填。
- ❌ **空泛的客户工作。** "省时间"太抽象。"出差期间提交英国公司季度增值税"才是工作。具体性才能露出真痛点。
- ❌ **痛点数量和痛点缓解数量严重失衡。** 8 个痛点配 1 个缓解,你没有价值主张,只有一个功能。
- ❌ **痛点缓解 / 收益创造 写成形容词。** "更快" / "更简单" —— 比什么快?怎么个简单法?换成机制描述。
- ❌ **混淆客户工作和痛点。** 工作是 _他们想做的事_;痛点是 _挡路的东西_。"逾期申报罚款"是痛点,不是工作。

## 语气

客户档案里的便签要像客户自己会说的话,不像营销 deck。价值地图的便签要描述机制,不要描述氛围。

## Blocks

`zoneId` 是 API 的稳定标识符,`pingarden canvas write` payload 必须用它,不要翻译。

### `gain-creators` — 收益创造方案

**Prompt** — 你的产品如何创造客户重视的具体收益？必要时用 Bain 价值元素标注。

**Example** — 每周节省 4 小时的手工对账

**Quality bar** — 收益创造方案描述你的产品及服务如何创造客户收益。 它明确描述了你打算提供给你客户的期望或使客户感到惊讶的结果及效益。 它包括功能效用、成本节约、积极情感。 由于有痛点缓释方案的存在，收益创造方案无须提及在客户概况中识别出的每一种收益。

### `pain-relievers` — 痛点解决方案

**Prompt** — 你的产品如何缓解一个具体客户痛点？

**Example** — 免去系统间的手工录入

**Quality bar** — 痛点缓释方案描述你的产品和服务如何减轻特定客户的痛点，明确描述你如何避免或减少客户在完成一项工作时的烦心事。 好的价值主张总是关注客户最重要的，特别是最极端的痛点。 对于你在客户概况中识别出的每一种痛点，你未必都有必要想出相应的缓释方案。

### `products-and-services` — 产品与服务

**Prompt** — 客户实际购买、使用或接触到什么？

**Example** — iOS 和 Android 移动应用

**Quality bar** — 这仅是一份你所提供的服务或产品清单。 这就像是客户在橱窗中所能看到的所有商品。 这是罗列你的价值主张所基于的所有产品和服务项目。 这些产品和服务能够帮助客户完成功能性、社会性或情感性工作，或能够帮助他们满足基本需求。

### `gains` — 收益

**Prompt** — 客户期待、渴望或会惊喜的结果是什么？可用 Bain 标签区分功能、情感、社会价值。

**Example** — 比竞争对手更快完成月结

**Quality bar** — 收益描述客户想要的结果或效益。 有些收益是客户所需要、期望或渴望的，有些是令他们惊讶的。 收益包括功能效用、社会收益、积极情绪及费用节省。

### `pains` — 痛点

**Prompt** — 客户在完成任务时痛在哪里？

**Example** — 每周五要花 6 小时手工核对数据

**Quality bar** — 痛点是指妨碍客户完成工作或客户在完成工作过程中所产生的。 痛点也是风险，换言之是潜在的不良结果与工作不能很好地完成有关。 试图将客户痛点划分为三类并且确认其严重度：不想要的结果、问题及特性；障碍；风险。

### `customer-jobs` — 客户工作

**Prompt** — 客户正在试图完成什么任务？

**Example** — 准确完成月度结账

**Quality bar** — 工作描述了你的客户在工作或生活中正尽力完成的事项。 客户工作可能是客户正在进行或尽力完成的工作、正尽力解决的问题或他们正尽力要去满足的需求。 在进行此项调查时，请务必从客户的角度进行分析。

## Pairs with

这张画布常和这些一起用,做完后引导用户接下去做:

- `business-model-canvas` — 把验证过的价值主张 + 客户细分搬回 BMC,让模型其余部分跟着改。
- `empathy-map` — 在写工作和痛点之前,先把客户在想 / 说 / 感受什么搞清楚。给右侧定锚。
- `jobs-to-be-done` — 先把客户工作锐化成「情境 → 动机 → 结果」故事，再用 Bain 标签分类价值层级。
- `ad-lib-value-proposition` — 把整张 VPC 压缩成一句团队能朗读出口的话 —— 弱点立刻暴露。
- `customer-journey` — 把痛点和收益沿着客户的时间线铺开,看清每一项在什么节点起作用。
- `design-criteria-canvas` — 把验证过的痛点和收益翻译成产品设计的硬约束。
- `experiment-canvas` — 挑出最不靠谱的那条痛点缓解或收益创造,为它设计最便宜的实验。
