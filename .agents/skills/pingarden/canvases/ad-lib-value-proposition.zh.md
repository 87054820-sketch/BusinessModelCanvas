# 一句话价值主张模板

## 何时使用

当你需要把一整张 VPC **压缩成一句团队能朗读出来的话**,用 Ad-Lib 价值主张速写。它是一种压力测试:如果填不顺,你的 VPC 一定有弱点 —— 通常是模糊的客户细分、被写成功能的价值主张、或者缺失的差异化。

这是一张句子填空式画布:每个区是一个槽,填完拼成一句:

> "我们的 **\<产品与服务>** 帮助 **\<客户细分>**(他们想 **\<工作>**),通过 **\<动词>** **\<痛点>** 并 **\<动词>** **\<收益>**,不同于 **\<竞争性价值主张>**。"

## 何时不该用

- 作为价值主张的起点。永远先做 **VPC**;Ad-Lib 在 VPC 下游。
- 多细分定位。一张 Ad-Lib 对应一个细分,跟 VPC 一样。
- 当作营销文案 / 广告标语。Ad-Lib 是对内的 —— 它的任务是暴露弱点,不是发布给客户看。

## 填写顺序 —— 一个槽一个槽,然后读出声

1. **`customer-segment`** —— 来自 BMC / VPC 的结构性细分。不是 persona,不是人口统计。
2. **`jobs-to-be-done`** —— VPC 里最重要的客户工作。带情感和社会层也行,只要一行内能写完。
3. **`customer-pain`** —— VPC 里最严重的痛点。
4. **`pain-verb`** —— 你的提供物如何作用于痛点:"减少 / 消除 / 避免 / 杜绝"。选**诚实**的动词,不是最强的。
5. **`customer-gain`** —— VPC 里最期望的收益。
6. **`gain-verb`** —— 你的提供物如何产生收益:"赋能 / 提升 / 实现 / 放大"。
7. **`products-services`** —— 真正的提供物(不是价值)。和 VPC 的产品与服务一致。
8. **`competing-value-proposition`** —— 客户今天在用什么(不只是直接竞品 —— 非消费 + 临时方案都算)。"不同于" 这一从句。

把整句读出声。读两遍。如果听起来别扭、什么新东西都没说、或者竞争对手也能用同样的话来说自己 —— 你的 VPC 不完整。

## 跨模块一致性

- **`customer-segment`** + **`jobs-to-be-done`** + **`customer-pain`** + **`customer-gain`** 必须全来自**同一张** VPC。这里混细分整张就废了。
- **`pain-verb`** 必须诚实反映强度:"消除"是最强主张,"减少"是有保留,"避免"是绕开。诚实选。
- **`competing-value-proposition`** 要具体到能点名。"其他工具"太空。说出那个临时方案。
- 整句话读出来应该是**一行**。如果摊成一段,各槽内容就太长了。

## 反模式 —— 不允许

- ❌ **没做 VPC 就填 Ad-Lib。** 你会写出听起来合理的小说。永远先 VPC,再压缩。
- ❌ **用"和 / 或"糊弄。** "帮助初创 _或_ 中型企业" 是两个细分。拆成两张 Ad-Lib。
- ❌ **`competing-value-proposition` = "没有好的替代品"。** 客户**今天**在做某事。找出来。Excel、实习生、忽略这问题 —— 都是真实的竞争对手。
- ❌ **撒谎的动词。** 实际只能"减少",却写"消除",这是红旗。Ad-Lib 是为了对内诚实。
- ❌ **`products-services` 写成结果。** 结果归到痛点 / 收益里。产品是真正的盒子 / app / 服务。
- ❌ **当成营销文案。** Ad-Lib 故意没有节奏感和文采。要标语,下游再做。

## 语气

死死压缩。每个词都得带信息量。如果竞争对手把自己产品名换上去这句话依然成立,你的差异化没立住 —— 改输入,不是改措辞。

## Blocks

`zoneId` 是 API 的稳定标识符,`pingarden canvas write` payload 必须用它,不要翻译。

### `products-services` — 产品与服务

**Prompt** — 句式中的具体产品/服务是什么？

**Example** — 实时协作白板

**Quality bar** — 写下客户真正会购买、使用或接触到的候选产品/服务。 越具体越好 ——「带时间点恢复的托管 PostgreSQL」比「数据基础设施」更适合进入一句价值主张。 这个槽位把句子锚定到客户真的能选择的东西。

### `customer-segment` — 客户细分

**Prompt** — 这句话服务哪个具体客户细分？

**Example** — A 轮创业公司的 CTO

**Quality bar** — 本句式只针对一个具体客户细分。 不要笼统地说「企业」或「用户」—— 要精确到行业、规模、岗位、场景或行为。 不同细分应写成不同速写句。 这个槽位防止价值主张变成泛泛而谈。

### `jobs-to-be-done` — 客户任务

**Prompt** — 这个客户想完成什么具体任务？

**Example** — 把产品规格无返工地交付给研发

**Quality bar** — 客户雇用某个方案要完成的任务，可以是功能性、社会性或情感性的。 用「动词 + 结果」表达：「按时交付一个功能」，而不是「需要一个工具」。 这个槽位解释客户想取得的进展，而不是你的解决方案。 它应把访谈证据或 JTBD 故事压缩成一个短结果。

### `pain-verb` — 动词(如减少、避免)

**Prompt** — 用哪个动词说明你如何缓解痛点？

**Example** — 减少

**Quality bar** — 选择一个说明你如何缓解痛点的动词，例如减少、消除、避免、降低。 它必须能自然连接右侧的具体客户痛点。 这个动词说明产品如何改变痛点，而不只是承认痛点存在。 它应该连接到 VPC 中具体的 Pain Reliever。

### `customer-pain` — 客户痛点

**Prompt** — 你要减少或避免哪个具体痛点？

**Example** — 数据库非计划停机带来的成本

**Quality bar** — 客户当下能切身感受到、具体且可观察的困扰。 优先选择强度高、频率高、会影响选择或切换行为的痛点，最好能被衡量。 这个槽位给价值主张提供紧迫性。 它应命名客户已经认可的障碍、风险、坏结果或摩擦，而不是团队凭空发明的问题。

### `gain-verb` — 动词(如提升、启用)

**Prompt** — 用哪个动词说明你如何创造收益？

**Example** — 提升

**Quality bar** — 选择一个说明你如何创造收益的动词，例如提升、启用、加速、增加。 它必须能自然连接右侧的具体客户收益。 收益动词解释产品让什么正向变化成为可能。

### `customer-gain` — 客户收益

**Prompt** — 你要增加或实现哪个具体收益？

**Example** — 5 分钟内回滚生产环境

**Quality bar** — 客户最终获得的具体好处，可以是更好结果、节省时间/成本、降低风险、提升形象或积极情绪。 写成可衡量或有场景感的结果。 这个槽位说明价值主张成立时客户获得什么。 它应描述客户想要的结果，而不是卖方内部指标。

### `competing-value-proposition` — 竞争价值主张

**Prompt** — 客户今天使用什么替代方案？

**Example** — 在 EC2 上自管的 Postgres

**Quality bar** — 写出客户今天已经在用的替代方案：直接竞品、手工变通、内部流程，或什么都不做。 正是这种对比让价值主张具备差异化和「为什么要切换」的理由。 这个槽位让价值主张进入比较语境。 客户永远有替代方案，即使它只是表格、手工流程或延后决策。
