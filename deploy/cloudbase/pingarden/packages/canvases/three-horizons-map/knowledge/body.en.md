# How to use

1. **Define the portfolio unit.** Decide whether each sticky represents a business unit, product line, platform, geography, or venture. Do not mix altitudes.
2. **Place H1 first.** List the current core businesses that create revenue, margin, data, distribution, or capabilities today.
3. **Place H2 next.** Add emerging businesses that already have evidence but are not yet core.
4. **Place H3 last.** Add future options that are promising but still mostly assumptions.
5. **Name migration criteria.** For each H3 or H2 item, specify what evidence, capability, funding, or governance change would move it to the next horizon.
6. **Record risks and evidence.** Put the unresolved assumptions in `Evidence and risks`; do not hide uncertainty inside the horizon columns.
7. **Translate into actions.** Decide fund, protect, scale, transfer, pivot, pause, or retire.

# How to read the map

A healthy Three Horizons Map has enough H1 to fund the company, enough H2 to create visible next growth, and enough H3 to keep future options alive. The key information is not only which column an item sits in. The key information is whether the item has a credible path to move.

Use `portfolio-map` when you need risk/return placement. Use Three Horizons when you need time sequencing, funding rhythm, and organizational handoff.

# Quality checklist

- Every item uses the same level of granularity.
- H1 contains real businesses, not vague capabilities.
- H2 has evidence, not only executive enthusiasm.
- H3 has learning goals, not science-fiction slogans.
- Each migration action names an owner, trigger, and next review.
- H2/H3 assumptions link to experiments or evidence scorecards.

# Common mistakes

- Treating H1, H2, H3 as calendar years. They are maturity horizons, not fixed dates.
- Filling H3 with fashionable technologies without customer or business-model hypotheses.
- Starving H2 because H1 metrics dominate all funding decisions.
- Calling a project H2 when it has no repeatable evidence.
- Forgetting the organizational handoff from exploration teams to operating teams.
- Using the map once and never tracking movement.
