# Smoke Test (烟雾测试)

一个看起来真实的落地页,推广一个你*还没造出来*的产品或功能。把目标流量打过来,数有多少人 **真正动手**(注册、"获取价格"、加入候补、点 "Buy")。Validation 阶段最经典的第一手 —— 在你 Discover 出有希望的方向之后,投工程之前要的量化需求证据。

## 什么时候用

- Discovery 已经完成 —— 访谈 + 问卷 + 论坛都指向同一个未被满足的需求。
- 在投工程之前,你需要比 "用户说会想要" 更强的证据。
- 团队有写文案 + 做设计 + 拉流量的能力(付费广告或自然种子)。
- 决策 gate 是 "要不要让工程团队上?" —— Smoke Test 是合适的 gate。

不要用:

- Discovery 都还没做 —— 会 smoke-test 错的 promise。
- 关系驱动 / 长周期的 B2B(落地页测不出真实意向)。改 Letter of Intent。
- 方案要解释、30 秒滚动里说不完。改 Concierge / Wizard of Oz。

## 怎么跑

1. **单一、锐利的价值主张。** 一个标题、一个副标、一个 CTA。这一页测的是 *promise*,不是产品。
2. **细节要像真的。** 定价(即使暂定的)、截图(mock 的也行但要具体)、能挣到的社证。
3. **一个 CTA。** "提前体验" / "加候补" / "获取价格"。多个 CTA 会让数据浑浊。
4. **拉 500-2000 个目标访问。** 对你 ICP 投付费广告、一篇 HN / PH / 即刻发帖、合作伙伴 newsletter。无目标流量就是噪音。
5. **守 promise。** 收邮箱,然后诚实告诉登记人产品在开发中,留一个跟进时段。在这里烧信任会毒到后面的实验。
6. **跑之前先定阈值** —— 比如 "目标访问者中 ≥ 8% 注册视为验证通过"。事后再调 = 没测。

## 什么算做得好

- 500-2000 个目标访问。
- 相对预设阈值有明确的转化率结果。
- 转化路径(哪条广告 → 哪个文案变体)可识别,可复制。
- 一份说"是"的人的干净注册名单 —— 他们就是 Customer Interview / Concierge 的招募对象。

## 反模式

- ❌ 伪造产品。不要显示一个吃卡但默默失败的结账。诚实地写"我们在开发,要不要提前体验"才行。
- ❌ 无目标流量。PH 首页带来的是好奇点击者,不是你的细分。
- ❌ 事后改阈值来宣布成功。
- ❌ 把邮箱名单当已验证需求,没有跟进步骤。很多人注册只为"看看"。
